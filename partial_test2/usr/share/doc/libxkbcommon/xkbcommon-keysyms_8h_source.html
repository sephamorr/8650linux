56 /*
57  * The "X11 Window System Protocol" standard defines in Appendix A the
58  * keysym codes. These 29-bit integer values identify characters or
59  * functions associated with each key (e.g., via the visible
60  * engraving) of a keyboard layout. This file assigns mnemonic macro
61  * names for these keysyms.
62  *
63  * This file is also compiled (by src/util/makekeys.c in libX11) into
64  * hash tables that can be accessed with X11 library functions such as
65  * XStringToKeysym() and XKeysymToString().
66  *
67  * Where a keysym corresponds one-to-one to an ISO 10646 / Unicode
68  * character, this is noted in a comment that provides both the U+xxxx
69  * Unicode position, as well as the official Unicode name of the
70  * character.
71  *
72  * Where the correspondence is either not one-to-one or semantically
73  * unclear, the Unicode position and name are enclosed in
74  * parentheses. Such legacy keysyms should be considered deprecated
75  * and are not recommended for use in future keyboard mappings.
76  *
77  * For any future extension of the keysyms with characters already
78  * found in ISO 10646 / Unicode, the following algorithm shall be
79  * used. The new keysym code position will simply be the character's
80  * Unicode number plus 0x01000000. The keysym values in the range
81  * 0x01000100 to 0x0110ffff are reserved to represent Unicode
82  * characters in the range U+0100 to U+10FFFF.
83  *
84  * While most newer Unicode-based X11 clients do already accept
85  * Unicode-mapped keysyms in the range 0x01000100 to 0x0110ffff, it
86  * will remain necessary for clients -- in the interest of
87  * compatibility with existing servers -- to also understand the
88  * existing legacy keysym values in the range 0x0100 to 0x20ff.
89  *
90  * Where several mnemonic names are defined for the same keysym in this
91  * file, all but the first one listed should be considered deprecated.
92  *
93  * Mnemonic names for keysyms are defined in this file with lines
94  * that match one of these Perl regular expressions:
95  *
96  * /^\#define XKB_KEY_([a-zA-Z_0-9]+)\s+0x([0-9a-f]+)\s*\/\* U+([0-9A-F]{4,6}) (.*) \*\/\s*$/
97  * /^\#define XKB_KEY_([a-zA-Z_0-9]+)\s+0x([0-9a-f]+)\s*\/\*\(U+([0-9A-F]{4,6}) (.*)\)\*\/\s*$/
98  * /^\#define XKB_KEY_([a-zA-Z_0-9]+)\s+0x([0-9a-f]+)\s*(\/\*\s*(.*)\s*\*\/)?\s*$/
99  *
100  * Before adding new keysyms, please do consider the following: In
101  * addition to the keysym names defined in this file, the
102  * XStringToKeysym() and XKeysymToString() functions will also handle
103  * any keysym string of the form "U0020" to "U007E" and "U00A0" to
104  * "U10FFFF" for all possible Unicode characters. In other words,
105  * every possible Unicode character has already a keysym string
106  * defined algorithmically, even if it is not listed here. Therefore,
107  * defining an additional keysym macro is only necessary where a
108  * non-hexadecimal mnemonic name is needed, or where the new keysym
109  * does not represent any existing Unicode character.
110  *
111  * When adding new keysyms to this file, do not forget to also update the
112  * following as needed:
113  *
114  * - the mappings in src/KeyBind.c in the repo
115  * git://anongit.freedesktop.org/xorg/lib/libX11.git
116  *
117  * - the protocol specification in specs/keysyms.xml

119  *
